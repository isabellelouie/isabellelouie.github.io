SAP

Activity Center: Monitoring Background Processes

Designing a consistent pattern for tracking background processes across SAP's analytics products.

OVERVIEW

Timeline

January 2026 – April 2026

Team

1 Backend Developer, 1 Frontend Developer, 1 Product Manager

Role

Led the design of the Activity Center pattern across SAP Analytics Cloud, SAP Datasphere, and SAP Business Data Cloud, defining the core interaction model for background process tracking.

Impact

Establishes a shared background process tracker across SAC, DSP, and BDC, replacing three divergent per-product implementations.

CONTEXT

SAP’s UX Consistency standards set hard constraints on shared UI patterns

SAP’s UX Consistency (UXC) standards define how common UI elements and patterns must appear and behave across all SAP products and lines of business. They cover shared elements such as Notifications, the Shell Bar, Side Navigation, and User Settings, ensuring users encounter familiar, predictable patterns as they move between products.

For Data & Analytics products like SAP Analytics Cloud, SAP Datasphere, and SAP Business Data Cloud. These products are often used together as part of a larger data workflow, so this consistency is especially critical. Users moving between products rely on shared UI conventions to orient themselves, and any divergence creates friction at the seams.

THE PROBLEM

Tackling a usability issue and a compliance gap

Problem 1: Background processes were buried in Notifications

Customer feedback pointed to a consistent usability problem: the Notifications panel had become too cluttered to use. Background processes (imports, exports, downloads, and calendar edits) were surfaced alongside standard user notifications. With 100+ unique notification types across application areas in each product, the panel became bloated and difficult to scan.

"I had 900+ notifications at a point. So there is no way that I can find the relevant things."

— SAP Analytics Cloud user feedback, September 2025

"Notifications are useless, there are too many, and no way to indicate priority."

— SAP Analytics Cloud user feedback, December 2025


Problem 2: Each feature team handled background processes differently

The UI patterns were also inconsistent across products. Each feature team had implemented their own approach to progress indicators, state changes, and notification copy, with no shared standard.



Problem 3: Background processes violated the UXC notification standard

UXC defines notifications as static messages: once sent, the content must not change. Background processes violated this by design, as their titles, descriptions, and states update continuously as the process runs.

THE SOLUTION

Introducing the Activity Center

The Activity Center is a dedicated panel in the shell bar where users can view and track all background processes in SAC, DSP, and BDC. Each process appears as a list item that updates in place through five states: queued, in progress, completed, completed with warnings, and failed.

Users can open the Activity Center at any time without disrupting their current workflow. Background processes are removed from the Notifications panel entirely, bringing SAC and DSP into compliance with the UXC notification standard.

KEY DESIGN DECISIONS

Showing Progress Without a Progress Bar

Working within a Design System constraint that ruled out the most obvious approach.

The most intuitive way to communicate task progress is a visual progress bar. SAP's Design System explicitly prohibits the NativeUI5 Progress Indicator component for running or ongoing processes, which ruled that out entirely.

OPTION A: STATIC STATE LABELS ONLY

One approach was to display only static state labels per item ("Queued", "In Progress", "Completed", "Failed") with no dynamic content while a task is running. This is fully compliant and technically lightweight, but provides minimal information during the most critical moment: when a task is actively running and a user wants to know how far along it is.

OPTION B: PROGRESS VIA DESCRIPTION TEXT (CHOSEN)

Instead, I used the description field to surface dynamic progress text ("5 of 11 objects processed") that updates as the task runs. This communicates meaningful progress without using a prohibited component, and has the added benefit of working better with screen readers than fractional notation.

The tradeoff is that text-based progress is less visually immediate than a bar. But given the constraint, it was the strongest option that kept the pattern compliant and accessible.

Resolving Naming Conflicts Across Three Products

Finding terminology that was unambiguous in SAP's existing ecosystem.

Naming the panel required checking it against existing SAP terminology, not just industry conventions. Terms commonly used for this pattern in other enterprise SaaS products ("Task Panel," "System Monitor," "Activity") each carried conflicting meaning in SAP's context. "Task Center" is already an SAP product for workflow task management. "Monitoring" is already an established feature in DSP, where it refers to data integration and replication task monitoring, a different scope entirely.

I evaluated naming options against two criteria: clarity for users moving between products, and zero overlap with existing features or product names across SAC, DSP, and BDC. "Activity Center" was the strongest option. It communicates ongoing background activity without implying user-assigned tasks ("Task Center"), system administration ("System Monitor"), or data flow performance metrics ("Monitoring").

Selecting the Shell Bar Icon

Ruling out overlap before committing to a symbol.

Adding a new icon to the shell bar required more than choosing one that looked right. I audited the icons already in use across SAC, DSP, and BDC to ensure the new icon would not be confused with an existing feature in any of the three products.

I also evaluated semantic alignment: the icon needed to communicate "something is running in the background" in a way that matched how users think about pending or ongoing work, and that differentiated clearly from the Notifications bell sitting immediately to its left.

After cross-referencing existing icon usage and reviewing the SAP icon library, I selected sap-icons://pending. It communicates "something in progress" without overlapping any existing shell feature across the three products, and it reads as meaningfully distinct from the Notifications icon.

Suppressing the Queued State to Prevent UI Flicker

A threshold rule for a state that usually lasts less than a second.

In most cases, background processes start almost immediately after being triggered. Showing a Queued state that transitions to In Progress within one or two seconds would cause the entry to visually flicker, appearing briefly in one state before jumping to another.

I defined a threshold: the Queued state should only be shown if it will persist longer than approximately 2–3 seconds. If a process starts immediately, the entry begins at In Progress. This keeps the panel stable and prevents state changes that add noise without adding information.

Updating Entries In Place

Keeping the list clean across a full process lifecycle.

Each background process entry updates within the same list item as its state changes, rather than creating a new item per state. Appending a new entry at each transition would cause the panel to fill with duplicates during active use, making it harder to follow any individual process.

In-place updates keep the list proportional to the number of active processes, not the number of state changes. Only the title, description, timestamp, and importance tag change as a process moves from queued to completed.

IMPACT

Shipping in Q4 2026 across SAC, DSP, and BDC

The Activity Center is currently in development and scheduled to ship across all three analytics products in Q4 2026. It will replace the inconsistent mix of notification patterns currently used across products with a single, shared interaction model.

Once live, it will give users a persistent place to track background tasks, reduce uncertainty during long-running operations, and establish a cross-product standard that product teams can build on as new process types are introduced.

Rollout and Governance

Shipping the component is only part of the work. Because each feature team writes its own activity center message copy independently, consistency depends on teams following the messaging guidelines I established alongside the component specs.

The rollout runs on two tracks. Externally, customers will be introduced through a What's New dialog and an external blog post at launch. Internally, the PM will distribute a What's New report to product managers across all three products. I will present the Activity Center at DNA Exchange (SAP's global Data & Analytics design knowledge sharing forum) to ensure design, development, UA, and PM teams across the organization are aligned on both the component and the messaging framework before teams begin adopting it.

REFLECTION

Cross-product alignment requires more than agreement. It requires a shared definition of the problem.

The hardest part of this project wasn't the component design. It was getting three product teams, each with their own shell implementation and workflow priorities, to agree on what "consistent" actually meant in practice. The surface pattern looks simple, but the alignment work underneath it was substantial.

Working within the design system's constraints also sharpened how I think about problem-solving. SAP's design system prohibits progress indicators for ongoing processes, which ruled out the most obvious solution. Rather than falling back on static state labels alone, I used the description field to surface dynamic progress text that updates in real time. The constraint forced a solution that ended up working better for screen readers than a visual bar would have.
