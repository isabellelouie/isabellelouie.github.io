SAP

Activity Center: Monitoring Background Processes

Designing a consistent pattern for tracking background processes across SAP's analytics products.

OVERVIEW

Timeline

January 2025 – Present

Team

1 Backend Developer, 1 Frontend Developer, 2 Product Managers

Role

Led the design of the Activity Center pattern across SAP Analytics Cloud, SAP Datasphere, and SAP Business Data Cloud, defining the core interaction model for background process tracking.

Impact

The Activity Center establishes a standardized pattern for background process tracking across SAP's analytics products, giving users clear visibility into system activity and reducing uncertainty during long-running operations.

CONTEXT

All SAP products must comply with a set of UX Consistency standards

SAP’s UX Consistency (UXC) standards define how common UI elements and patterns should appear and behave across all SAP products and lines of business.

These standards include shared elements such as Notifications, the Shell Bar, Side Navigation, and User Settings. Standardizing these patterns ensures users encounter familiar, predictable patterns as they move between SAP products.

For Data & Analytics products like SAP Analytics Cloud, SAP Datasphere, and SAP Business Data Cloud – which are often used together as part of a larger data workflow – this foundational consistency is especially important for users whose journey spans multiple products.

THE PROBLEM

Background processes were in the wrong place and handled inconsistently across products

In SAC and DSP, background processes — imports, exports, downloads, and calendar edits — were surfaced inside the Notifications panel alongside standard user notifications. With 100+ unique notification messages across all application areas within each product, the panel became bloated and difficult to scan.

The UI patterns for these processes were also inconsistent. In some features, the notification title changed dynamically as the process progressed; in others it stayed static. Some used a progress bar; others indicated progress with dynamic text. There was no standard for notification copy either — each feature team wrote their own message text, leading to inconsistencies in punctuation, file name formatting, and phrasing across process types.

These processes also violated the UXC definition of notifications, which requires notification content to remain static once sent. Beyond compliance, this created three user-facing problems:

CHALLENGE #1: BACKGROUND PROCESSES GET BURIED

Background processes can take several minutes to complete. In that time, other notifications from across the product continue to accumulate, pushing the process further down the panel and making it hard to find.

CHALLENGE #2: INCONSISTENT PATTERNS ACROSS PRODUCTS

SAC, DSP, and BDC each surfaced background processes differently, creating different expectations and behaviors depending on which product a user was working in.

CHALLENGE #3: INTERRUPTED WORKFLOWS

Without persistent status, users either had to wait in place or navigate away and lose track of what was running.

THE SOLUTION

Introducing the Activity Center

The Activity Center is a dedicated panel in the application shell where users can view and track all background processes across SAC, DSP, and BDC. Each process appears as a list item that updates in place through five states: queued, in progress, completed, completed with warnings, and failed.

Per UXC-016, product-specific shell actions must appear immediately to the right of the Notifications icon. I placed the Activity Center entry point in that position across all three products using a consistent shell bar icon, giving users a predictable entry point without disrupting the existing shell layout.

Users can open the Activity Center at any time without disrupting their current workflow. Background processes are removed from the Notifications panel entirely, bringing SAC and DSP into compliance with the UXC notification standard.

KEY DESIGN DECISIONS

A deep dive into the decisions that shaped the pattern

Showing Progress Without a Progress Bar

Working within a Design System constraint that ruled out the most obvious approach.

The most intuitive way to communicate task progress is a visual progress bar. SAP's Design System explicitly prohibits the NativeUI5 Progress Indicator component for running or ongoing processes, which ruled that out entirely.

OPTION A: STATIC STATE LABELS ONLY

One approach was to display only static state labels per item ("Queued", "In Progress", "Completed", "Failed") with no dynamic content while a task is running. This is fully compliant and technically lightweight, but provides minimal information during the most critical moment — when a task is actively running and a user wants to know how far along it is.

OPTION B: PROGRESS VIA DESCRIPTION TEXT (CHOSEN)

Instead, I used the description field to surface dynamic progress text ("5 of 11 objects processed") that updates as the task runs. This communicates meaningful progress without using a prohibited component, and has the added benefit of working better with screen readers than fractional notation.

The tradeoff is that text-based progress is less visually immediate than a bar. But given the constraint, it was the strongest option that kept the pattern compliant and accessible.

Designing for Cross-Product Consistency

One pattern, three products with different shells.

SAC, DSP, and BDC each have different shell implementations and product-specific workflows. The Activity Center pattern needed to work across all three without requiring each product team to build a custom solution.

I worked with product and engineering teams across all three products to define a shared interaction model: a consistent entry point, consistent status states, and a consistent list structure. Product teams then implement the pattern within their own shell without diverging from the core behavior.

Selecting the Shell Bar Icon

Ruling out overlap before committing to a symbol.

Adding a new icon to the shell bar required more than choosing one that looked right. I audited the icons already in use across SAC, DSP, and BDC to ensure the new icon would not be confused with an existing feature in any of the three products.

I also evaluated semantic alignment — the icon needed to communicate "something is running in the background" in a way that matched how users think about pending or ongoing work, and that differentiated clearly from the Notifications bell sitting immediately to its left.

After cross-referencing existing icon usage and reviewing the SAP icon library, I selected sap-icons://pending. It communicates "something in progress" without overlapping any existing shell feature across the three products, and it reads as meaningfully distinct from the Notifications icon.

Suppressing the Queued State to Prevent UI Flicker

A threshold rule for a state that usually lasts less than a second.

In most cases, background processes start almost immediately after being triggered. Showing a Queued state that transitions to In Progress within one or two seconds would cause the entry to visually flicker — appearing briefly in one state before jumping to another.

I defined a threshold: the Queued state should only be shown if it will persist longer than approximately 2–3 seconds. If a process starts immediately, the entry begins at In Progress. This keeps the panel stable and prevents state changes that add noise without adding information.

Updating Entries In Place

Keeping the list clean across a full process lifecycle.

Each background process entry updates within the same list item as its state changes, rather than creating a new item per state. The alternative — appending a new entry at each transition — would cause the panel to fill with duplicates during active use, making it harder to follow any individual process.

In-place updates keep the list proportional to the number of active processes, not the number of state changes. Only the title, description, timestamp, and importance tag change as a process moves from queued to completed.

IMPACT

Shipping in Q4 2026 across SAC, DSP, and BDC

The Activity Center is currently in development and scheduled to ship across all three analytics products in Q4 2026. It will replace the inconsistent mix of notification patterns currently used across products with a single, shared interaction model.

Once live, it will give users a persistent place to track background tasks, reduce uncertainty during long-running operations, and establish a cross-product standard that product teams can build on as new process types are introduced.

REFLECTION

Cross-product alignment requires more than agreement — it requires a shared definition of the problem

The hardest part of this project wasn't the component design. It was getting three product teams, each with their own shell implementation and workflow priorities, to agree on what "consistent" actually meant in practice. The surface pattern looks simple, but the alignment work underneath it was substantial.

Working within the Notifications UI framework also sharpened how I think about design constraints. Rather than treating the framework as a ceiling, I treated it as a set of guardrails and looked for the best solution within them. That mindset led to a stronger outcome than simply defaulting to what the framework made easy.
