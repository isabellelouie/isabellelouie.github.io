Flair Airlines: Cross-Platform Design System

Modernizing Flair Airlines' design system to support mobile-first experiences and scalable UI across web and mobile platforms.

OVERVIEW

Timeline

2 months (May - June 2025)

Role

As a UX/UI Designer at Flair Airlines, I evolved the design system to introduce a modern, responsive and scalable foundation across mobile app and web experiences.

Outcome

The updated design system launched on the Flair Airlines mobile app in March 2026.

INTRODUCTION

Designing for a Mobile-First Future

As part of Flair Airlines' broader effort to modernize its digital products, I evolved the design system to support a mobile-first experience and align with the airline's refreshed brand direction.

KEY INSIGHT

Over 60% of Flair Airlines' traffic came from mobile devices!

THE PROBLEM

Challenges with the Previous System

The original design system had been optimized for desktop. Many components relied on horizontal layouts, excessive white space, and large typography that did not translate well to smaller screens.

The system also used a complex colour palette, where multiple accent colours could appear within a single screen, making it difficult to establish a clear visual hierarchy.

When scaled to mobile, these components resulted in inefficient layouts, excessive scrolling, and reduced information density.

PROCESS

Rebuilding from the ground up (pun intended)

Rather than patching individual components, I started with the layer below them. A component system is only as consistent as the tokens it’s built on. Getting the tokens right first meant everything built on top would be consistent by default.

Step 1: Token Audit and Rebuild

Replacing hardcoded values with a structured variable system.

I audited the existing Figma file to identify every hardcoded color, type style, and spacing value. Many components used raw hex values instead of variables, which meant a single brand change required manually updating dozens of components.

I rebuilt the token system from scratch, aligning it to Flair’s updated brand direction: a simplified color palette with clear hierarchy, a mobile-first typography scale, and a spacing system that worked across breakpoints.

Key Decision: Simplifying the Color Hierarchy

The previous palette used multiple purple and green variations within a single screen, which flattened visual hierarchy and made interactive elements harder to identify. I restructured the color roles: dark grey as the primary UI color, green as the secondary accent, and purple reserved for highlight states only. This reduced visual noise and aligned the interface more closely with the updated brand.


Step 2: Mobile-First Component Architecture

Designing smallest-first, then scaling up.

With the token system in place, I built approximately 20 core components and the full booking flow. The first decision was how to approach the build.

Key Design Decision: Adapt Desktop Components vs. Start from Mobile

The existing components were built desktop-first. The straightforward path was to adapt them by adding responsive breakpoints and adjusting sizing.

OPTION A: ADAPT DESKTOP COMPONENTS FOR MOBILE

Add responsive breakpoints and scale down existing components. Faster to start, but the underlying assumptions (horizontal layouts, large whitespace, desktop interaction patterns) stay baked into the structure.

OPTION B: DESIGN MOBILE-FIRST, SCALE UP (CHOSEN)

Start every component from the smallest screen size, using mobile constraints as the design baseline: touch targets, vertical layouts, compact spacing. Then scale up to larger breakpoints from there.

Starting from mobile constraints produced components that worked natively on the primary platform. Scaling up from a mobile base is structurally cleaner than retrofitting desktop components for touch. The spacing, hierarchy, and interaction patterns are correct from the start rather than adjusted after the fact.


Step 3: Handoff Documentation

Writing specs that could stand on their own.

Developers implemented from my Figma specs into Storybook. To ensure accuracy without real-time back-and-forth, I built documentation directly into the design system file: interaction behavior annotations, token references, spacing specifications, and edge case notes for each component.

The goal was specs clear enough that a developer could implement without needing to ask follow-up questions. Each component page referenced the relevant tokens so values would pull from the system rather than being hard-coded at implementation.

IMPACT

A foundation built to ship.

From token rebuild to live product.

The rebuilt system covered approximately 20 core components and the full booking flow, all built on a structured token system aligned to the updated brand. The system launched on the Flair Airlines mobile app in March 2026.

~20 COMPONENTS BUILT

Buttons, checkbox, radio button, switch, tag, accordion, card, countdown, list, message strip, message note, tab bar, seat map, footer, content section, hero, modal, navigation, and payment form fields, plus the full booking flow.

TOKEN SYSTEM REBUILT FROM SCRATCH

Color, typography, and spacing tokens replaced a system of hardcoded values, aligning the interface to the updated brand and making future changes maintainable.

LAUNCHED MARCH 2026

The updated system shipped on the Flair Airlines mobile app and continues to support the development of new digital experiences.

REFLECTION

What I would do differently

The value of early developer involvement.

The biggest limitation of this project was the timing of developer involvement. Implementation happened after the design work was complete, which meant there was no feedback loop during the build phase to catch constraints early.

Having a developer involved earlier — even informally — would have surfaced implementation edge cases during component design rather than after handoff. For a system meant to ship to production, design and development feedback loops matter as much as the component architecture itself. The specs I wrote were thorough, but a two-way conversation would have made them better.
